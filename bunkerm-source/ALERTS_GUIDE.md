# BunkerM Alerts Guide

BunkerM includes a built-in notification system that surfaces **broker health alerts** and **AI-detected anomalies** directly in the dashboard header.

---

## How Alerts Appear

Click the **bell icon** (🔔) in the top-right header to open the alert panel.

- A red badge shows the number of **unacknowledged critical or high** alerts.
- Clicking **Ack** dismisses an individual alert.
- Clicking the refresh icon re-fetches alerts immediately.

**Role visibility:**

| Alert category | admin | user |
|---|---|---|
| Broker alerts | ✅ visible | ❌ hidden |
| Anomaly alerts | ✅ visible | ✅ visible |

---

## Broker Alerts

Broker alerts are generated by the monitor service every time `/api/v1/stats` is polled (approximately every 30 seconds from the frontend).

| Alert | Friendly title | Severity | Trigger condition | Default threshold | Environment variable |
|---|---|---|---|---|---|
| **Broker unreachable** | Broker Unreachable | 🔴 critical | Broker has not published `$SYS` data for N consecutive polls | 3 polls (~30 s) | `ALERT_BROKER_DOWN_GRACE_POLLS` |
| **High latency** | High Latency | 🟠 high | Round-trip ping latency exceeds threshold | 500 ms | `ALERT_LATENCY_HIGH_MS` |
| **Client capacity** | Client Capacity Warning | 🟠 high | Connected clients exceed X% of max_connections | 80% of max | `ALERT_CLIENT_CAPACITY_PCT`, `ALERT_CLIENT_MAX_DEFAULT` |
| **Message spike** | Message Spike Detected | 🟠 high | Incoming message rate (1 min) exceeds N× recent baseline | 5× baseline | `ALERT_MSG_SPIKE_FACTOR` |
| **In-flight high** | In-flight Messages High | 🟠 high | Messages awaiting acknowledgement exceed threshold | 500 messages | `ALERT_INFLIGHT_MAX` |
| **Reconnect loop** | Client Reconnect Loop | 🟠 high | A single client reconnects N times within a time window | 5 connects / 60 s | `ALERT_RECONNECT_LOOP_COUNT`, `ALERT_RECONNECT_LOOP_WINDOW_S` |
| **Auth failures** | Authentication Failures | 🟠 high | Mosquitto authentication failures exceed threshold within a time window | 5 failures / 60 s | `ALERT_AUTH_FAIL_COUNT`, `ALERT_AUTH_FAIL_WINDOW_S` |

### Changing Thresholds

All thresholds are configured via environment variables in `docker-compose.yml` under the `bunkerm` service. The defaults are shown with the `:-` syntax, for example:

```yaml
- ALERT_LATENCY_HIGH_MS=${ALERT_LATENCY_HIGH_MS:-500}
```

To override, add the variable to a `.env` file next to `docker-compose.yml`:

```env
ALERT_LATENCY_HIGH_MS=250
ALERT_CLIENT_CAPACITY_PCT=90
ALERT_MSG_SPIKE_FACTOR=3
```

Then redeploy: `.\deploy.ps1 -Action patch-backend`

## Anomaly Detection Alerts

The Smart Anomaly Detection service (`ai` service, port 8100) monitors MQTT topic values over time using two statistical methods:

| Algorithm | Friendly label | What it means |
|---|---|---|
| **Z-score** | Unusual Reading | A single value is far outside the historical distribution (e.g. a temperature sensor suddenly reports 150°C) |
| **EWMA** | Sudden Trend Change | The recent moving average has shifted significantly compared to the longer-term average (gradual drift or step change) |

### Severity Levels

Severity is determined by how many standard deviations above the threshold the anomaly score is:

| Score above threshold (σ) | Severity |
|---|---|
| < σ + 1 | 🔵 low |
| σ + 1 | 🟡 medium |
| σ + 2 | 🟠 high |
| σ + 3 | 🔴 critical |

The base sigma threshold is configured via:

```yaml
- ANOMALY_SIGMA_THRESHOLD=${ANOMALY_SIGMA_THRESHOLD:-3.0}
```

Increasing this value makes anomaly detection less sensitive (fewer false positives). Decreasing it makes it more sensitive.

### Acknowledgement Workflow

1. An anomaly alert appears in the notification panel under **Anomalies**.
2. Click **Ack** to acknowledge it — it is marked as acknowledged in the database and disappears from the panel.
3. A **cooldown window** (default: 5 minutes) prevents the same alert from firing again immediately after acknowledgement.

Acknowledged alerts are stored in SQLite (`/nextjs/data/smart-anomaly.db`) and are not shown again unless a new anomaly event occurs for the same topic.

## Architecture Overview

```
Mosquitto ($SYS topics)
       │
       ▼
Monitor Service (port 1001)
  ├── AlertEngine (in-memory, evaluates every stats poll)
  │     └── GET /api/v1/alerts/broker
  │         POST /api/v1/alerts/broker/{id}/acknowledge
  └── MQTT stats → frontend dashboard charts

Smart Anomaly Service (port 8100)
  ├── Detects Z-score and EWMA anomalies per MQTT topic
  ├── Stores alerts in SQLite
  └── GET /api/v1/alerts
      POST /api/v1/alerts/{id}/acknowledge

Next.js Frontend
  ├── useNotifications hook (polls both services)
  └── UpdateNotification component (bell icon, role-based)
```

## Disabling Specific Alerts

To disable an alert type, set its threshold to an unreachable value:

```env
# Effectively disable message spike alerts
ALERT_MSG_SPIKE_FACTOR=99999

# Effectively disable latency alerts
ALERT_LATENCY_HIGH_MS=999999
```

To disable anomaly detection entirely, stop or remove the `ai` service from `docker-compose.yml`.

## Troubleshooting

**Bell icon shows no alerts even though the broker is down**  
→ The broker down alert fires after `ALERT_BROKER_DOWN_GRACE_POLLS` consecutive polls. The first poll failure is ignored to avoid false positives on restarts.

**Anomaly alerts not appearing**  
→ The Smart Anomaly service needs at least 1 hour of data before it can establish a baseline. Ensure the `ai` service is running and MQTT topics are publishing data.

**Alerts reappear after acknowledging them**  
→ Broker alerts (in-memory) will re-appear if the condition is still true on the next stats poll. Anomaly alerts observe a 5-minute cooldown per topic/type.
